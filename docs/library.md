# Библиотека пайплайнов SciGuide

## Назначение

Библиотека SciGuide предоставляет единое ядро пайплайнов для обработки корпуса и поиска по нему.

Она используется в двух сценариях:

- интеграция в backend-сервис;
- работа в Jupyter Notebook для исследований, экспериментов и замеров качества.

Библиотека использует:

- `Qdrant` как векторное хранилище;
- `Neo4j` как графовое хранилище;
- `LangChain` как основу для построения пайплайнов.

Пользователь библиотеки работает с одной точкой входа: `PipelineManager`.

## Точка входа

`PipelineManager` является основным пользовательским объектом библиотеки.

Он отвечает за:

- инициализацию клиента библиотеки;
- открытие подключений;
- управление подключениями в течение жизненного цикла;
- доступ к пайплайну нарезки;
- доступ к пайплайну поиска.

Пользователь работает с библиотекой через менеджер и не управляет подключениями к `Qdrant` и `Neo4j` вручную в стандартном сценарии.

## Входные данные

Библиотека использует единый контракт входных данных для сервиса и Jupyter Notebook.

На вход пайплайнов передается содержимое файла, полученное через `f.read()`, и связанные с ним метаданные.

Это означает:

- в Jupyter Notebook файл читается обычным способом через `open(...).read()` или `with open(...) as f: f.read()`;
- в backend-сервисе входные данные приводятся к объекту `io`, чтобы библиотека получала тот же тип содержимого, что и в ноутбуке;
- библиотека работает с содержимым файла и метаданными, а не с объектами, специфичными для web-фреймворка или интерфейса загрузки.

Такой контракт делает сценарии сервиса и ноутбука одинаковыми с точки зрения пользовательского API.

## Быстрый старт для приложения

Стандартный сценарий интеграции в сервис строится вокруг минимального количества действий:

1. создать `PipelineManager`;
2. передать конфигурацию подключений и моделей;
3. вызвать нужный пайплайн;
4. использовать результат в логике приложения;
5. корректно завершить жизненный цикл менеджера.

Пример пользовательского сценария:

```python
from sciguide_pipeline import PipelineManager

manager = PipelineManager(config)

chunk_result = manager.chunking.run(documents)
search_result = manager.search.run(query="Что такое graph-guided retrieval?")
```

Для прикладного разработчика этого достаточно, чтобы встроить библиотеку в сервис без ручной сборки цепочек и без отдельного управления каждым хранилищем.

## Работа в Jupyter Notebook

В Jupyter Notebook используется тот же `PipelineManager`, что и в сервисе.

Это позволяет:

- работать с тем же пользовательским API;
- получать сопоставимые результаты;
- проводить эксперименты на той же логике пайплайнов;
- изменять состав и параметры пайплайнов для замеров качества.

Пример пользовательского сценария:

```python
from sciguide_pipeline import PipelineManager

manager = PipelineManager(config)

manager.chunking.run(documents)
result = manager.search.run("Методы графового перерангирования в RAG")
result.items
```

В исследовательском режиме пользователь может модифицировать пайплайны, чтобы сравнивать разные стратегии обработки корпуса и поиска, не выходя за рамки общей библиотеки.

## Конфигурация

Для работы библиотеки требуется конфигурация, достаточная для запуска пайплайнов и подключения внешних зависимостей.

Конфигурация должна описывать:

- подключение к `Qdrant`;
- подключение к `Neo4j`;
- настройки моделей и цепочек `LangChain`;
- параметры пайплайна нарезки;
- параметры пайплайна поиска.

Конфигурация задается на уровне `PipelineManager` и используется всеми пайплайнами, если пользователь явно не переопределяет отдельные параметры сценария.

## Жизненный цикл

Пользователь работает с менеджером как с объектом, который владеет жизненным циклом инфраструктурных подключений.

Ожидаемое поведение:

- при создании менеджера библиотека подготавливает клиент и подключения;
- во время работы пайплайны используют уже открытые соединения;
- после завершения работы ресурсы корректно освобождаются.

Это позволяет одинаково удобно использовать библиотеку и в долгоживущем сервисе, и в коротких исследовательских сессиях.

## Обязательные зависимости

Библиотека рассчитана на сценарий, в котором:

- векторное представление хранится в `Qdrant`;
- графовая модель хранится в `Neo4j`;
- пайплайны строятся на `LangChain`.

Эти компоненты являются частью пользовательского контракта библиотеки.
