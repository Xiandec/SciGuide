# PRD / описание проекта SciGuide

## 1. Название проекта

**SciGuide** — автоматическая справочная система по выбранному научному направлению на основе языковых моделей, семантического поиска и графового перерангирования.

## 2. Контекст и проблема

При работе с научными материалами пользователь сталкивается с двумя типовыми проблемами. Во-первых, классический keyword search плохо находит фрагменты, где нужная идея выражена другими формулировками. Во-вторых, даже семантический поиск по эмбеддингам не всегда учитывает структурные связи между понятиями: какие термины часто встречаются вместе, какие темы логически следуют друг за другом, какие концепты образуют устойчивые локальные кластеры в корпусе.

Из-за этого в top-k выдаче могут оказываться семантически похожие, но контекстно слабые фрагменты, тогда как более полезные куски текста теряются, если их релевантность проявляется не только через близость к запросу, но и через положение в сети понятий предметной области. Для научной справочной системы это особенно критично, потому что ответы часто требуют не одного фрагмента, а согласованного набора связанных фрагментов.

## 3. Цель проекта

Разработать прототип автоматической справочной системы, которая отвечает на вопросы по выбранному научному направлению, используя двухэтапный retrieval pipeline:

1. первичный поиск релевантных текстовых чанков по векторному семантическому сходству;
2. дополнительное перерангирование найденных кандидатов с учетом графа связей между понятиями, извлеченными из корпуса.

## 4. Идея решения

Базовый слой системы — обычный semantic search по эмбеддингам документов и пользовательского запроса. Он дает хороший recall и позволяет быстро получить ограниченный набор кандидатов.

Поверх него строится графовая модель корпуса. На текущем этапе извлечение сущностей в пайплайне нарезки выполняется без LLM, без внешней NER-модели и без внешней онтологии. Для каждого текстового фрагмента детерминированно генерируются n-граммы длиной до четырех токенов. Для каждой candidate-фразы считается score: базовый вес зависит от длины n-граммы, дополнительные бонусы добавляются за числа и дефисы. Затем кандидаты сортируются, дубликаты и вложенные варианты отфильтровываются, и для каждого чанка остается ограниченный список сущностей.

При индексации документа обновляются служебные структуры графа:

- `doc_to_entities` — список сущностей, найденных в документе;
- `entity_to_docs` — список документов, в которых встречалась сущность;
- `token_to_entities` — индекс токенов сущностей для неполного совпадения;
- `entity_tokens` — токенизированное представление сущности.

Каждая сущность становится узлом графа. Ориентированные ребра добавляются по локальному порядку сущностей внутри чанка: для каждой сущности рассматриваются следующие одна-две сущности в списке, и между ними создается ребро `left_entity -> right_entity`. Если такое ребро уже существует, его вес увеличивается. Таким образом, граф в прототипе — это ориентированный граф совместного появления и порядка терминов, а не RDF и не академический knowledge graph.

Далее запрос тоже проецируется в пространство сущностей. Сначала для него детерминированно извлекаются опорные сущности, затем графовый поиск использует не только точные совпадения строк, но и пересечение токенов через индекс `token_to_entities`. После этого учитываются близкие узлы по ориентированным ребрам, и полученный structural signal используется для перерасчета score чанков. В итоге итоговая релевантность зависит не только от близости эмбеддингов, но и от того, насколько чанк покрывает важные узлы и связи в окрестности запроса.

Такой подход соответствует современному направлению graph-guided retrieval: граф не заменяет векторный поиск, а уточняет его, добавляя relational signal.
