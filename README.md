# SciGuide

Курсовая работа по теме «Разработка автоматической справочной системы на основе языковых моделей для выбранного научного направления».

SciGuide — реализованный в рамках этой работы прототип автоматической справочной системы для работы с собственным научным корпусом: пользователь загружает материалы в отдельный воркспейс и получает ответы, опирающиеся на документы этого корпуса, а не на абстрактные знания модели.

## О проекте

Ключевая идея SciGuide в том, что для научных текстов недостаточно обычного keyword search и недостаточно "чистого" семантического поиска по эмбеддингам. В научном корпусе релевантность часто определяется не только похожими формулировками, но и связями между понятиями, соседством терминов и их местом в локальной структуре предметной области.

Поэтому SciGuide использует двухэтапный retrieval:

1. первичный поиск релевантных фрагментов по векторному сходству;
2. дополнительное графовое перерангирование по связям между извлеченными сущностями.

На практике это дает систему, которая помогает:

- собирать рабочую базу знаний по конкретной теме внутри отдельного воркспейса;
- задавать вопросы к собственному корпусу документов, а не к внешнему интернет-контексту;
- получать ответы на основе найденных фрагментов, связанных не только семантически, но и структурно;
- разделять разные темы, наборы документов и рабочие контексты между воркспейсами.

Иными словами, SciGuide — это не просто интерфейс к LLM, а прикладной RAG-сервис для научных материалов с отдельным retrieval-слоем как центральной частью проекта.

## Quick Start

### Требования

- Docker
- Docker Compose

### Запуск

```bash
cp .env.example .env
docker compose up --build -d
```

После запуска будут доступны:

- приложение: `http://localhost`
- backend healthcheck: `http://localhost/health`
- Swagger UI backend: `http://localhost:8000/docs`
- MinIO console: `http://localhost:9001`
- Neo4j browser: `http://localhost:7474`

### Что проверить перед стартом

- в `.env` должен быть задан `JWT_SECRET_KEY`;
- для полноценных ответов языковой модели нужно указать `OPENROUTER_API_KEY`;
- при первом запуске сервис может подниматься дольше обычного, потому что скачиваются embedding- и reranker-модели в `hf_cache/`.

### Базовый сценарий проверки

1. Открой `http://localhost`.
2. Зарегистрируй пользователя.
3. Создай воркспейс.
4. Задай системный промпт и загрузи документ.
5. Дождись фоновой обработки документа.
6. Создай чат и отправь вопрос по материалам воркспейса.

## Как это устроено

На верхнем уровне проект состоит из нескольких частей:

- `frontend` на Vue 3: интерфейс входа, списка воркспейсов, настроек, документов и чатов;
- `backend` на FastAPI: API, бизнес-логика, аутентификация, работа с воркспейсами и сообщениями;
- `Celery`-worker: фоновая индексация документов;
- `sciguide_pipeline`: общая библиотека пайплайнов нарезки, поиска и графового перерангирования;
- инфраструктурные хранилища: `PostgreSQL`, `Redis`, `MinIO`, `Qdrant`, `Neo4j`;
- `Nginx`: единая точка входа в приложение.

Коротко поток обработки выглядит так:

- документ загружается в воркспейс и сохраняется в `MinIO`;
- backend ставит задачу на индексацию;
- worker запускает библиотеку `sciguide_pipeline`;
- чанки и эмбеддинги попадают в `Qdrant`, граф сущностей — в `Neo4j`;
- при сообщении в чат backend поднимает контекст из документов воркспейса, добавляет системный промпт и историю диалога, после чего формирует ответ.
